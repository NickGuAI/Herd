# Daemon And iOS Rollout Guide

Use this guide when you need to decide whether a machine can run provider
processes through the Herd daemon, or when you need to build and run the
iOS wrapper locally. It separates code availability from live rollout state.

## Distribution State

```text
╔══════════════════════════════════════════════════════════════════╗
║ Herd Distribution State                                    ║
╠══════════════════════════════════════════════════════════════════╣
║ Daemon code exists?                  yes                       ║
║ Daemon paired on current machines?   check with daemon-status   ║
║ Daemon auto-start/service?           not generated by CLI       ║
║ iOS wrapper exists?                  yes, Capacitor             ║
║ iOS local build path?                pnpm cap:build + Xcode     ║
║ TestFlight/App Store scaffold?       checked in, manual upload  ║
╚══════════════════════════════════════════════════════════════════╝
```

The daemon is transport and process plumbing only. The Herd server remains
canonical for commanders, conversations, approvals, transcripts, quests,
telemetry, and UI replay. Pairing a daemon changes how provider CLI processes
run on a machine; it does not move server-owned state to that machine.

## Daemon Runtime Model

```text
┌──────────────────────────────┐
│ Herd server             │
│ sessions, quests, approvals  │
│ transcripts, telemetry, UI   │
└──────────────┬───────────────┘
               │ /api/agents/daemons/ws
               │ pairing-token websocket
┌──────────────▼───────────────┐
│ herd daemon run         │
│ provider health + process IO │
└──────────────┬───────────────┘
               │ local spawn
┌──────────────▼───────────────┐
│ claude / codex / gemini      │
│ host-local provider auth     │
└──────────────────────────────┘
```

The codebase includes the daemon protocol, registry, websocket upgrade handler,
pair/status/revoke APIs, CLI daemon runner, provider health reporting, and
daemon-backed provider spawn paths. A live machine is still SSH-backed until it
is explicitly paired and connected.

## Check Current Deployment

List registered machines:

```bash
herd machine list
```

Check one machine's daemon rollout state:

```bash
herd machine daemon-status --machine <id>
```

Interpret the output this way:

| Field | Meaning |
| --- | --- |
| `Transport` | Selected process transport for this machine: `local`, `ssh`, or `daemon`. |
| `Paired` | The server has a daemon pairing record for this machine. |
| `Connected` | A live `herd daemon run` websocket is attached now. |
| `Provider auth` | The daemon reported whether provider CLIs are authenticated. |
| `Launchable` | Server can launch provider sessions through the daemon now. |
| `Providers` | Per-provider installed/authenticated/version details from the daemon. |

If `Transport` is `ssh`, or `Paired`/`Connected` is `no`, the machine is not yet
running through daemon transport even though the daemon code exists.

## Pair And Run A Daemon

Create or refresh the server-side pairing:

```bash
herd machine daemon-pair --machine <id>
```

The CLI prints the exact command to run on the target machine:

```bash
herd daemon run --machine <id> --pairing-token <token> --endpoint <url>
```

Run that command on the target machine and keep it alive. The current CLI does
not generate a daemon-specific launchd or systemd service. If an operator needs
auto-start, wrap the printed `herd daemon run ...` command in host-local
service management and keep the pairing token protected.

Re-check readiness after the daemon connects:

```bash
herd machine daemon-status --machine <id>
```

The machine is ready for daemon-backed launches when `Connected: yes`, Provider
auth is `ready`, and `Launchable: yes`. If provider auth is not ready, authenticate
the relevant provider CLI on that host, then re-run `daemon-status`.

Revoke a daemon pairing when the token should no longer be usable:

```bash
herd machine daemon-revoke --machine <id>
```

Existing live machines may still report SSH transport until an operator runs
the pairing flow above.

## iOS Wrapper Status

The iOS app lives under `apps/herd/ios` and is a Capacitor wrapper around
the Vite-built web app. It has no separate native business-logic tree. It uses
the first-run Connect screen to store the selected Herd/Herd instance
URL and scoped API key; native fetch and websocket calls use that stored
instance. The hosted `https://herd.gehirn.ai` URL is the default suggestion,
not a hard-coded backend.

Server-side CORS must allow the Capacitor origin:

```bash
export HERD_ALLOWED_ORIGINS="https://herd.gehirn.ai,capacitor://localhost"
```

For self-hosted deployments, add `capacitor://localhost` to that instance's
allow-list and restart the server.

Authentication on iOS is currently API-key based unless native Auth0 redirect
handling is repaired. Create a scoped API key through the target instance's
**/api-keys** page, or generate a one-time mobile pairing invite from
**Settings -> Mobile Access** and scan or paste that invite on the iOS Connect
screen.

## Build And Run iOS Locally

Prerequisites:

- macOS with Xcode 15 or newer.
- Apple ID signed into Xcode.
- Apple Developer team membership for device/archive work.
- Repo dependencies installed with `pnpm install --frozen-lockfile`.

Build and open the Xcode project:

```bash
cd apps/herd
pnpm cap:build
pnpm cap:ios
```

In Xcode, select a simulator or connected device and run. For the unattended
CLI build/install path, signing details, and troubleshooting, use
[iOS Build Guide](ios-build-guide.md).

## Distribution Status

There is no checked-in Fastlane or GitHub Actions release pipeline for the
current iOS wrapper. The repository does include App Store Connect export
scaffolding (`ios/App/exportOptions.plist`), the privacy manifest, and the
manual archive/export/upload sequence in [iOS Build Guide](ios-build-guide.md).

For TestFlight or App Store distribution, an operator still needs paid
membership in team `YQYSGND73G`, a distribution signing setup on
`yus-mac-mini`, an App Store Connect app record for `ai.gehirn.herd`, and
the actual upload/TestFlight install evidence. Do not treat the local
`pnpm cap:build` path as a production distribution pipeline.
